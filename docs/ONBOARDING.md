# How a Developer Creates a Service

This guide walks through the end-to-end process of creating a new service using the HSF "Create a Service" workflow. The process takes about 2-3 minutes and requires no Harness UI interaction after submitting the form.

## Prerequisites

Before you begin, confirm the following:

1. **IDP access.** You have access to the Harness Internal Developer Portal and can see the Workflows section.
2. **Target project exists.** The Harness project where your service will live must already exist. Available projects include:
   - Solutions Factory (`Harness_Platform_Management`)
   - Image Factory (`Harness_Platform_Management`)
   - Core (`Platform`)
   - Sandbox (`Playground`)
   - Default Project (`default`)
3. **Session 1 deployed.** The `trigger_workspace_on_git_file_change` pipeline must be deployed in the HSF project. This is handled by the platform team.
4. **Session 2 deployed.** The `create_service` pipeline must be deployed. Run `bootstrap/deploy.sh` if it has not been.

## Step-by-Step: Creating a Service

### Step 1: Open the IDP Workflow

Navigate to: **IDP > Workflows > Create a Service**

You will see a form with two fields.

### Step 2: Fill in the Form

| Field | Description | Rules | Example |
|-------|-------------|-------|---------|
| **Service name** | The name of your service. This becomes the GitHub repo name. | Lowercase, alphanumeric, hyphens allowed. Must start and end with alphanumeric. | `my-api-service` |
| **Target Harness project** | The Harness project where triggers and input sets will be created. | Select from the dropdown. | `Solutions_Factory` |

### Step 3: Submit

Click **Create**. The IDP workflow triggers the `create_service` pipeline in the HSF project.

### Step 4: Wait for Completion

The pipeline runs four stages sequentially. Total time is typically 2-3 minutes:

| Stage | What Happens | Duration |
|-------|-------------|----------|
| Scaffold Repo | Creates GitHub repo, seeds `triggers.yaml` and `Dockerfile` | ~10s |
| Create Harness Service | Creates a service resource in the target project | ~5s |
| Create Workspace A | Creates IACM workspace, applies to generate initial triggers | ~60-90s |
| Wire Auto-Apply | Calls Session 1's pipeline to set up auto-reconciliation | ~60-90s |

You can watch progress by clicking the "View pipeline execution" link in the IDP output.

## What Gets Created

After the pipeline succeeds, the following resources exist:

### GitHub

- **Repository:** `https://github.com/Kru3ish/<service_name>`
  - `README.md` (auto-generated by GitHub)
  - `.harness/triggers.yaml` (default single-service config)
  - `Dockerfile` (minimal Alpine-based stub)
- **Webhook:** Registered on the repo, pointing to Harness. Fires on push to `main` when `.harness/triggers.yaml` changes.

### Harness

- **Service:** `<service_name>` (identifier: hyphens replaced with underscores) in your target project.
- **Input set:** `input_<service_name>_all` in the target project, referencing your CI pipeline.
- **Push trigger:** `push_<service_name>_all` -- fires on merge to `main`.
- **PR trigger:** `pr_<service_name>_all` -- fires on pull request open/reopen/sync.
- **Workspace A:** `service_triggers_<service_name>` in the HSF project -- manages triggers from `triggers.yaml`.
- **Workspace B:** `watcher_service_triggers_<service_name>_<service_name>` in the HSF project -- watches for `triggers.yaml` changes.

### Tags on All Resources

```
managed_by: hsf
created_via: create_service_workflow
service_name: <your-service-name>
```

## How to Verify Everything Worked

### 1. Check the GitHub repo

Visit `https://github.com/Kru3ish/<service_name>` and confirm:
- The repo exists.
- `.harness/triggers.yaml` contains a single trigger entry with `path: "*"`.
- A `Dockerfile` exists.

### 2. Check the Harness service

In the Harness UI, navigate to your target project > Services. Confirm `<service_name>` appears.

### 3. Check triggers and input sets

In the Harness UI, navigate to your target project > Pipelines > select the CI pipeline > Triggers tab. Confirm:
- `push_<service_name>_all` trigger exists and is enabled.
- `pr_<service_name>_all` trigger exists and is enabled.

Navigate to Input Sets tab. Confirm:
- `input_<service_name>_all` input set exists.

### 4. Check IACM workspaces

In the Harness UI, navigate to `Harness_Platform_Management / Solutions_Factory` > IACM > Workspaces. Confirm:
- `service_triggers_<service_name>` exists with a successful apply.
- `watcher_service_triggers_<service_name>_<service_name>` exists.

### 5. Check the GitHub webhook

On GitHub, go to `Kru3ish/<service_name>` > Settings > Webhooks. Confirm a webhook pointing to `https://app.harness.io/gateway/ng/api/webhook?accountIdentifier=...` is registered.

## What Happens Next: Day-2 Changes

Once your service is created, any changes to `.harness/triggers.yaml` are automatically applied. See the [Monorepo Guide](./MONOREPO-GUIDE.md) for how to:

- Add monorepo path-based triggers.
- Add PR-only or push-only triggers.
- Remove trigger entries.

The workflow is: edit the file, open a PR, merge. Harness auto-reconciles within about 30 seconds.

## Adding Services to a Monorepo

If your repo is a monorepo (multiple services in one repo), you do not need to run the IDP form again. Instead, edit `.harness/triggers.yaml` to add entries for each service path. See [MONOREPO-GUIDE.md](./MONOREPO-GUIDE.md) for the full guide.
